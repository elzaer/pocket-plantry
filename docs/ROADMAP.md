# Pantry & meal planner — feature roadmap

This structures everything discussed into epics and user stories, roughly in build order. Epic 0 covers the product-resolution and data-model work already designed (barcode lookup chain, generic item / SKU mapping, checkout fulfilment) — it's the foundation everything else sits on.

---

## Epic 0 — Product resolution (foundation)

Already scoped in detail; listed here for completeness and sequencing.

- **Story:** As the app, when a barcode is scanned, resolve it through local cache → Open Food Facts → optional paid fallback → manual entry, so every product only needs resolving once.
- **Story:** As a user, when I manually enter a product, my entry (data + photo) is pushed back to Open Food Facts, growing the shared dataset.
- **Story:** As a user, every product I create gets mapped to a generic item (existing or new) at the point of entry, so it can later fulfil a shopping list requirement regardless of which SKU I actually buy.
- **Story:** As the app, at checkout, a scanned product resolves to its generic item, marks any matching open shopping list entry fulfilled, and always logs to pantry stock.

**Acceptance:** any barcode scan reliably ends in a known product linked to a generic item, with minimal repeat manual entry.

---

## Epic 1 — Weekly meal planning

- **Story:** As a user, I create a meal plan for the week made up of meals, where each meal lists generic item requirements (not specific products).
- **Story:** As a user, each ingredient defaults to "one unit of the matching generic item" with no quantity specified.
- **Story (future-proofing):** As a developer, ingredient rows carry an optional quantity field in the schema, unused by any logic today, so quantity-aware planning can be added later without a migration.

**Acceptance:** a saved meal plan can be reduced to a flat list of generic item IDs needed for the week.

---

## Epic 2 — Recurring requirements list

- **Story:** As a user, I maintain a list of generic items I buy regularly outside meal plans (household consumables, pantry staples).
- **Story:** As a user, I can add, remove, or edit items on this list at any time.
- **Stretch story:** As a user, I import roughly 3 months of past receipts and the app suggests recurring items to seed this list automatically, based on purchase frequency.

**Acceptance:** the requirements list behaves as just another source of generic item IDs feeding into shopping list generation — no special-casing needed downstream.

---

## Epic 3 — Pantry stock tracking

- **Story:** As a user, I can mark what's currently on hand at generic-item granularity (have it / don't have it — no pack-size precision, per the earlier scoping decision).
- **Story:** As the app, stock updates automatically whenever a checkout scan or receipt import resolves to a generic item (from Epic 0 / Epic 5).
- **Story:** As a user, I can do a manual "walk the pantry" pass to correct stock that's drifted out of sync (used something without scanning it, etc).

**Acceptance:** pantry stock reliably reflects has/doesn't-have for every generic item the household tracks.

---

## Epic 4 — Shopping list generation

- **Story:** As a user, my shopping list is generated by combining this week's meal plan requirements and the recurring requirements list, minus whatever's already in pantry stock.
- **Story:** As a user, each list entry displays my preferred product as the default suggestion, without locking me into buying that exact SKU.
- **Story:** As the app, list entries get marked fulfilled progressively as I scan purchases at checkout (Epic 0's checkout logic).

**Acceptance:** given a meal plan, requirements list, and current stock, the generated list matches what's actually still needed — no manual reconciliation.

---

## Epic 5 — Receipt import and put-away

- **Story:** As a user, I import a receipt from a shopping trip and the app parses it into line items.
- **Story:** As the app, each receipt line is matched to a known product where possible (barcode/SKU match).
- **Story:** As a user, for any receipt line that doesn't resolve to a known product, I'm prompted to do the manual-entry/mapping flow (Epic 0) while I'm unpacking groceries — same UI, just triggered from an import instead of a scan.
- **Story:** As the app, every resolved receipt line runs through the same checkout-fulfilment logic as a live scan (marks list items fulfilled, updates pantry stock).

**Acceptance:** after importing a receipt and finishing put-away prompts, the shopping list and pantry stock both reflect the actual trip with minimal manual re-entry.

---

## Epic 6 (stretch) — Predictive replenishment

- **Story:** As the app, I build a consumption history from receipt imports, meal plan usage, and manual pantry adjustments over time.
- **Story:** As the app, I estimate a rough purchase cadence per generic item (e.g. "peanut butter roughly every 3 weeks").
- **Story:** As a user, I see "likely due soon" suggestions surfaced onto the requirements list or shopping list automatically.

**Acceptance:** the system pre-fills a meaningful share of a typical week's shopping list without the user manually building it from scratch.

---

## Suggested build order

1. **Epic 0** — foundation, nothing else works end to end without it.
2. **Epic 3** (pantry stock, manual entry only) + **Epic 2** (requirements list, manual only) — these unblock list generation.
3. **Epic 1** (meal planning).
4. **Epic 4** (shopping list generation) — first genuinely useful, demoable milestone.
5. **Epic 5** (receipt import) — the biggest quality-of-life jump, but leans heavily on Epic 0's mapping UI, so build it after that's proven out with real manual-entry usage.
6. **Epic 2's receipt-based stretch, then Epic 6** — both need real accumulated receipt history to be worth anything, so there's a natural lag here regardless of how fast you build; don't front-load effort on these.

## Open questions to resolve as you go

- Which receipt input format(s) to support first — photo OCR, retailer CSV export, or manual paste — this scopes Epic 5's first story significantly.
- How much "recently used" weighting the Epic 0 mapping-UI autocomplete needs, since Epic 5's put-away flow will lean on it constantly.
- Whether the requirements list needs any cadence metadata before Epic 6, or whether it's fine to stay a flat list until prediction actually exists.
